<completed_work>

- **Critical eval bug found and fixed**: `events_from_targets` in validate.py used threshold >= 0.5, but onset targets are pre-widened in the dataset as [0.3, 0.6, 1.0, 0.6, 0.3]. This inflated reference events by 3x, making recall appear ~33% when frame-level recall was actually ~97-100%. Fixed to threshold >= 1.0.

- **Model outputs changed to logits**: Removed sigmoid from onset/offset heads in model forward pass. Loss now uses `binary_cross_entropy_with_logits` for numerical stability. Sigmoid applied at inference boundaries (validate.py). Velocity head still outputs [0,1] via sigmoid.

- **Loss function refactored**: `DrumscribbleLoss` now supports `pos_weight` (via native BCEWithLogitsLoss) and `target_widening` params, but neither should be used — plain BCE (pos_weight=1, no widening) works best.

- **Train/test split created**: `scripts/split_parquet.py` splits by `source_audio` to prevent data leakage. 90/10 split. Train: 148,536 rows, Test: 16,472 rows. Originals backed up to `features/_original/` in each dataset dir.

- **Diagnosis of pos_weight experiments**: pos_weight=5 and pos_weight=20 were both solving the wrong problem. They caused broad activations (8-9x false positive over-prediction) while the model already had near-perfect recall. All those training runs were wasted due to the eval bug.

- **Class coverage verified**: 22/26 classes active in training data. Missing: MIDI 35 (acoustic bass drum), 41 (low floor tom), 57 (crash cymbal 2), 77 (wood block).

- **musdb18hq downloaded**: ~/Documents/Datasets/musdb18hq — 150 tracks with isolated drum stems, but no onset annotations so can't use for automated eval. Could use for qualitative testing.

- **Schism repo explored**: ~/Code/Schism — Swift audio separation framework with 15+ CoreML models. Natural pipeline: Schism separates drum stem → DrumscribbleCNN transcribes to MIDI. Both target CoreML/ANE deployment.

<decisions_made>

- **Plain BCE is best**: pos_weight and target widening are counterproductive. The eval bug made it look like recall was low, but it was actually near-perfect. Standard BCE produces the best results.
- **Logits-based loss**: Model outputs logits for onset/offset, sigmoid applied only at inference. More numerically stable.
- **Split by source_audio**: Prevents leakage between augmented copies of the same recording.
- **22/26 classes is acceptable**: The 4 missing classes are niche instruments rarely found in typical drum recordings.

<context>
The big revelation this session was that the model was performing well the entire time — the evaluation methodology was broken. The pre-widened onset targets ([0.3, 0.6, 1.0, 0.6, 0.3] pattern in the dataset) meant every real onset generated 3 reference events at the 0.5 threshold. Peak-picking correctly collapsed predictions to 1 event per onset, so it looked like 33% recall when it was actually ~97%.

With the corrected evaluation on a proper test split, epoch 10 already shows F1=0.846 (precision=0.868, recall=0.825 at threshold 0.1). This should improve significantly over the remaining 90 epochs.

The next big milestone is getting true held-out numbers on the final model, then potentially integrating with Schism for a complete separation→transcription pipeline running entirely on-device via CoreML.
</context>
